## Overview

Fieldgrid’s billing model has two main layers:

- **Operational invoicing** – invoices you send to your customers for jobs and work orders.
- **Platform billing** – your own Fieldgrid subscription (plans, seats, modules, trials, payment status).

This guide covers both from the admin perspective.

## Entry points

- Operational invoicing:
  - `/admin/invoices` → invoice list and builder.
  - `/invoices/:id` → invoice detail view via `InvoiceRouter`.
- Platform billing:
  - `/billing/*` routes:
    - `/billing/upgrade`
    - `/billing/seats`
    - `/billing/modules`
    - `/billing/checkout/success`, `/billing/checkout/cancel`
    - `/billing/payment-required`
    - `/billing/trial-expired`

## Operational invoicing flow

### 1. Create an invoice

- From `/admin/invoices`:
  - Use **Create invoice** action to open the invoice builder.
- In the builder:
  - Select customer and project/job context.
  - Add line items:
    - Labor (time-based).
    - Materials (inventory items).
    - Fixed fees or discounts.
  - Configure taxes and totals.
- Data is validated with forms and sent to backend invoice APIs.

### 2. Review and send

- Use the preview tab to:
  - Review invoice layout.
  - Download or generate a PDF (via helpers like `utils/invoicePdfExport.ts`).
- Once finalized:
  - Mark as sent.
  - Optionally trigger email delivery via backend integrations.

### 3. Track status

- Back on `/admin/invoices`:
  - Filter by status (draft, sent, paid, overdue).
  - View links back to jobs and work orders.

## Platform billing flow

### 1. Trials and conversion

- When new organizations sign up:
  - They may start on a trial.
- Trial status is tracked in:
  - `billingStore` (`trialExpired`, `daysRemaining`, `showTrialWarning`).
  - `authStore` (via `setSubscriptionStatus`).
- When trials end:
  - Users may be redirected to `/billing/trial-expired`.
  - Critical features can be gated behind billing routes via `ProtectedRoute` and 402 handling in `api/client.ts`.

### 2. Upgrades, seats, and modules

- On `/billing/upgrade` and related pages:
  - Admins select plans, seat counts, and additional modules.
- `billingStore` provides helpers:
  - `fetchPricingPlans`, `fetchPremiumModules`, `updateSeats`, `addModule`, `cancelSubscription`, `hasModule`, `canAddTechnicianSeat`, etc.
- Actions here typically:
  - Redirect to a payment portal or Stripe checkout.
  - Update subscription state via backend billing APIs.

### 3. Payment-required and subscription issues

- The HTTP client in `src/api/client.ts` handles HTTP 402 responses:
  - For `trial_expired` reasons, it updates trial flags and redirects to `/billing/trial-expired`.
  - For `upgrade_plan` or other reasons, it:
    - Dispatches events.
    - Opens the **Upgrade** modal using `upgradeModalStore`.
- When `subscriptionStatus` is `past_due` or `canceled`:
  - `ProtectedRoute` may redirect to:
    - `/billing/payment-required`
    - `/billing/reactivate` (if present)
