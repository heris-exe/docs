## What this space is

This docs site is our **shared knowledge hub** for Fieldgrid:

- A single place to explain how features work, how workflows fit together, and how we talk about the product.
- Backed by GitHub, so changes move through the same review and versioning process as code.
- Designed for **everyone** on the team—not just developers.

## Who contributes what

Different teams naturally own different types of pages:

- **Product & Engineering**
  - Deep guides on workflows (for example, admin job lifecycle, technician daily workflow).
  - API reference and internal runbooks.
- **Sales & Success**
  - Role-based solution pages and FAQs that come up on calls.
  - Phrases and examples that land well with customers.
- **Design & Marketing**
  - Language around positioning and UX intent.
  - Visuals or diagrams that clarify complex flows.
- **Onboarding & Ops**
  - Getting-started guides and “first week” playbooks.
  - Internal checklists for rollouts and demos.

If you understand a part of the product or process well enough to explain it in Slack or a meeting, you’re qualified to add or edit docs here.

## How changes flow

At a high level, changes move like this:

1. **Capture the knowledge**
   - Turn a question, Slack thread, or “quick sync” into a short doc or FAQ entry.
2. **Edit in the Mintlify editor or code**
   - Make updates via the web editor, or edit the `.mdx` file in the docs repo.
3. **Review like code**
   - Open a PR when editing via Git; ask a teammate who knows the area to review.
4. **Ship and reuse**
   - Once merged, the new or updated page becomes the link we share in chats, tickets, and calls.

Over time, this keeps answers from being trapped in DMs or one‑off meetings.

## When to make a new page vs update an existing one

Use this simple rule of thumb:

- **Update an existing page** when:
  - You’re clarifying a step in a workflow guide.
  - You’re correcting language or screenshots.
  - You’re answering a question that clearly belongs in an existing FAQ or troubleshooting doc.
- **Create a new page** when:
  - You’re documenting a new workflow, role, or integration.
  - You need an internal runbook for a recurring issue.
  - You’re adding a reference that doesn’t fit cleanly anywhere else.

If you’re not sure, update an existing page first—maintainers can always split things out later.

## How this supports demos and workshops

When we run product demos or internal trainings, we aim to:

- Use **real docs pages** as the source of truth instead of ad‑hoc walkthroughs.
- Show different roles where **their** starting points live (for example, “Solutions for admins,” “Admin getting started,” “Technician getting started”).
- Capture new questions that come up and turn them into FAQs or troubleshooting entries afterward.

This way, every workshop or demo makes the docs stronger for the next person.
